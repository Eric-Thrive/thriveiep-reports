Agentic Coding Loop Engineering

Last-three-month research on agentic coding loops, with a loop-card playbook for DEC1TEXTREDACTOR, Accommodation Engine, and PM-Hub.

Process Reference June 26, 2026 · Research dive & implementation guide

Scope

This report covers community resources, official docs, and publications surfaced between 2026-03-26 and 2026-06-26, with prior PM-Hub research used as baseline context. The focus is not generic "AI agents"; it is the practical concept of loops in agentic coding: repeated work cycles where an agent acts, evidence is checked, state is persisted, and the next action is chosen from the measured result.

The recommendations are tailored to the repeated work patterns already visible in DEC1TEXTREDACTOR, Accommodation Engine, Bloom, BC/Post-Secondary report work, C2A/DDx content work, and PM-Hub's own research harness.

Bottom Line

The strongest pattern across the last three months is that loop quality matters more than agent autonomy. The best loops are small, evidence-bound systems.

  1. Define the objective as a measurable contract.
  2. Isolate each attempt in a safe work surface.
  3. Run one focused change.
  4. Verify with a trusted signal.
  5. Persist the trace, result, and decision.
  6. Promote reusable lessons only when they clear a governance rule.
  7. Stop when the evidence says success, saturation, or unsafe ambiguity.

For your work, the practical opportunity is to standardize a family of loop cards across DEC1TEXTREDACTOR and Accommodation Engine instead of treating each loop as bespoke chat craft. You already have the right raw ingredients: Karpathy-style F2 optimization in Redactor, bounded report pipelines in Bloom, multi-surface planning rules in PM-Hub, and skills such as local-review, plan-harden, autonomous-plan-implementer, arch-checkpoint, and multi-surface-agent-harness. The missing layer is a shared loop contract: trigger, metric, state file, verifier, memory promotion rule, stop condition, and replay command.

What A Loop Is In Agentic Coding

A loop is not "let the agent keep going." A loop is a controlled system:

objective -> attempt -> verification -> decision -> memory -> next attempt or stop

The key distinction is that a loop has a measurable decision point. Without that, you only have repeated prompting.

The current literature and docs converge on six necessary parts:

PartRequired QuestionWhy It Matters
ObjectiveWhat must be true at the end?Prevents drift and premature completion.
Verification surfaceWhat evidence proves progress or failure?Turns subjective agent confidence into a check.
Attempt boundaryWhat can change this iteration?Keeps blast radius small and diffs reviewable.
State artifactWhat gets persisted after each pass?Allows resume, audit, learning, and replay.
Memory ruleWhat becomes future guidance, and when?Prevents both forgetting and skill-library drift.
Stop conditionWhen should the loop complete, pause, or escalate?Prevents wasted iterations and unsafe autonomy.

Best Practices

1. Start With A Goal Contract, Not A Prompt

Use this shape for any loop you expect to repeat:

Achieve <desired end state>, verified by <specific evidence>,
while preserving <constraints>. Use <allowed inputs/tools/files>.
Between iterations, choose the next action by <selection rule>.
Stop if <blocked/saturation/unsafe condition> and report <evidence needed>.

For DEC1TEXTREDACTOR:

Improve composite PII F2 by at least 0.002, verified by the frozen benchmark
suite, while preserving recall >= 0.990 and no new high-risk FN class. Use only
T1-T4 recognizer/post-filter/config changes unless a holdout fixture exists.
Between iterations, prioritize the failure cluster with highest weighted F2
headroom. Stop after three non-improving valid attempts or any recall-floor
breach.

For Accommodation Engine report work:

Raise evidence-grounding pass rate for BC/Post-Secondary reports, verified by
claim-to-source checks and fixture snapshots, while preserving existing report
schema and customer overlay boundaries. Use only the bounded pipeline stages,
authority docs, and approved fixture set. Between iterations, fix the highest
severity verifier failure class first. Stop if authority sources conflict or a
surface lacks an owner.

2. Prefer Small Loops With Strong Checks Over Large Autonomous Loops

OpenAI's repair-loop cookbook uses Review, Repair, Validate. That pattern is useful because each phase has a different job. Review observes without editing. Repair changes a copied artifact. Validate produces structured feedback.

For your work, preserve that separation:

  • Redactor: diagnose failure cluster → apply one bounded detector/post-filter/config change → benchmark → keep/reject.
  • Bloom/BC reports: diagnose unsupported claim or renderer mismatch → patch one stage/contract → run fixture/verifier → keep/reject.
  • DDx content: diagnose banned phrase or voice collapse → edit rule/template → run lint/sample audit → keep/reject.
  • PM-Hub process: diagnose repeated agent failure → update AGENTS/skill/plan template → test on next live task → promote or revert.

3. Build Loops Around Proactivity Carefully

The proactivity paper's useful distinction is that proactive agents are judged by an insight policy, not by activity volume. For your workflows, a proactive loop should answer:

  • What changed?
  • Why does it matter now?
  • What evidence supports interrupting Eric?
  • What is the smallest useful next action?
  • What feedback updates the policy?

Good proactive loops:

  • detect stale BC authority docs before implementation;
  • detect Redactor benchmark drift after a new ground-truth update;
  • detect report-engine fixture regression after schema or renderer changes;
  • detect Linear drift when commits lack issue references;
  • detect skill-library drift when an often-invoked skill causes repeated fixes.

Bad proactive loops:

  • scan everything daily and produce unactionable summaries;
  • open issues without a severity threshold;
  • write to source-of-truth systems without a separate implementation plan;
  • keep repeating "review this" without new evidence.

4. Do Not Rely On Execute-Fail-Fix Alone

TEBench's warning maps directly to your environment. Standard coding agents are decent at fixing tests that already fail. They are poor at stale tests that pass but no longer assert the right behavior, and missing tests for new behavior.

Implementation consequence:

  • Every loop needs an "absence of failing tests is insufficient" check.
  • For Redactor, a benchmark pass must still ask whether a hidden FN category was not represented in the corpus.
  • For AE, a green unit suite must still ask whether customer authority, source attribution, and renderer/export boundaries are covered.
  • For BC/Post-Secondary, claim grounding must check missing evidence and stale authority, not just parser failures.

5. Treat Skills, AGENTS.md, And Rules As Runtime Artifacts

The last-three-month research makes this sharper:

  • Rules help, but guardrails beat guidance. Negative constraints are safer than positive style directives.
  • Skills are better than bloated global instruction files for procedures.
  • Trace-derived skills can help, but ungoverned accumulation causes library drift.

Recommended local policy:

  • Put short, stable, negative constraints in AGENTS.md.
  • Put multi-step procedures in skills.
  • Put per-loop runtime findings in loop state files first, not directly in AGENTS.md.
  • Promote a lesson to a skill only after it appears in at least two distinct loop traces or one P0/P1 review finding.
  • Retire or quarantine skills/rules that have not helped in the last N uses or that correlate with repeated misfires.

6. Use Deterministic Guardrails For Load-Bearing Boundaries

Do not ask an agent to "be careful" with secrets, PII, authority docs, or export contracts. Use hooks, scripts, schema validators, golden fixtures, and negative permissions.

Examples:

  • A Redactor loop should not be allowed to print raw student data or private document contents in logs.
  • A PM-Hub/Google Workspace loop should use gws with file-backed credentials, never browser scraping.
  • An AE report loop should not modify renderer output strings to hide upstream schema gaps.
  • A multi-surface loop should not implement against missing or conflicting authority docs; it should mark the card NEEDS-CLARIFICATION.
  • A secret-access hook should block plaintext secret retrieval into the context, following the same design logic as the AWS Secrets Manager agent guidance.

7. Preserve Fresh Review Context

For risky work, the reviewer should not be the same context that authored the change. This is already embedded in your local-review, plan-harden, and multi-surface harness practice. Keep it.

Use same-thread self-checks for cheap syntax and mechanical errors. Use fresh review for:

  • security/privacy changes;
  • schema migrations;
  • report/renderer/export boundary changes;
  • benchmark-driven tuning where overfitting risk is real;
  • agent skill/rule changes that affect future sessions;
  • any change that crosses authority surfaces.

Loop Types To Standardize

Type 1: Repair Loop

Use for broken tests, validation failures, stale docs, export errors, and schema mismatch.

review -> repair -> validate -> repeat until pass or blocker

Minimum artifacts: loop-card.md, attempts.jsonl, validation command, changed-file summary, final keep/reject decision.

Type 2: Optimization Loop

Use for Redactor F2 tuning, latency, cost, cache hit rate, evaluator accuracy, or report grounding rate.

hypothesis -> bounded edit -> metric run -> keep/reject -> memory update

Minimum artifacts: baseline metrics, metric weights and floors, allowed edit tiers, holdout rule, rejection memory keyed by structural target (not just hypothesis ID), rollback command.

Type 3: Review Loop

Use for human/agent review cycles.

diff -> review comments -> targeted follow-up -> smaller diff -> approval

Minimum artifacts: review scope, severity taxonomy, comments anchored to lines or sections, follow-up validation, unresolved risk list.

Type 4: Proactive Monitor Loop

Use for recurring health signals: stale authority docs, benchmark drift, Linear drift, docs/KG staleness, CRM hygiene, report index gaps.

watch signal -> classify importance -> gather evidence -> interrupt or log -> learn from feedback

Minimum artifacts: watch query/command, threshold, evidence packet, interrupt policy, "do not alert if" rules, feedback field.

Type 5: Skill/Memory Promotion Loop

Use when repeated corrections indicate a reusable instruction or procedure.

trace -> cluster failures -> propose rule/skill edit -> test on fixture/live case -> promote/retire

Minimum artifacts: source traces, proposed memory text, expected behavior change, evaluation case, expiry/review date, retirement trigger.

Type 6: Multi-Surface Implementation Loop

Use for report-engine, BC, assessment-bank, KG, source-attribution, and deployment-contract changes.

control artifact -> surface cards -> authority check -> implementation slices -> verification -> index/update

Minimum artifacts: one source-controlled control artifact, card per affected surface, authority status per card, out-of-scope notes, verification per slice, done-when criteria. This should continue to route through multi-surface-agent-harness.

Specific Opportunities

DEC1TEXTREDACTOR

Current pattern: seven-stage pipeline with GLM-OCR, Qwen LLM passes, Presidio, post-filter, user review, export; F2-weighted benchmark with recall pressure; Karpathy-style optimization history; architecture state/changelog; correction memory and failure clusters; recent Wave-C autonomous launch surfaced live loop-framework bugs.

Highest-ROI loop upgrades:

  1. Replace raw unified-diff planner output with structured edit operations. The Wave-C failure mode showed that planner-generated diffs are brittle. A safer edit schema is: 
    {
  "target_file": "server/config/detection-config.json",
  "operation": "replace_json_value",
  "path": ["ner_config", "thresholds", "PERSON"],
  "old_value": 0.72,
  "new_value": 0.74,
  "rationale": "Suppress low-confidence PERSON FPs in score-term contexts"
}
  2. Deduplicate rejection memory by structural key, not by hypothesis ID: 
    target_file + operation_type + affected_entity_type + normalized_diff_hash + failure_cluster
    This prevents T2-01-v2, T2-01-v3, and similar variants from burning iteration budget.
  3. Make tier selection part of the loop contract. Do not let the planner choose Tier 5 unless there is a holdout set and a user-approved prompt/model experiment window. Default to T1-T4 for routine accuracy tuning.
  4. Use dual metrics: primary composite F2; hard floors on recall by entity type and no new high-risk FN class; ratchet so precision improves only if recall floor is stable.
  5. Add proactive stale/missing-test checks. After every accepted change, ask: which class of FPs did this suppress? what class of FNs might it introduce? is that class represented in the benchmark? If not, create a pending ground-truth fixture request instead of trusting the metric.
  6. Maintain two memories: rejection memory (failed edits and why) and success memory (accepted edits with the smallest generalizable rule).
  7. Keep human gates at: any change touching OCR model behavior; any prompt/model change; any change that lowers recall floor; any ground-truth relabeling; any production data handling change.

Accommodation Engine / Bloom / BC Post-Secondary

Current pattern: Bloom already has a bounded staged report pipeline; BC/Post-Secondary is moving toward evidence-row and source-grounded review; report architecture framework already names durable execution, agent registry, citation verification, and learning evaluator; PM-Hub global rules require multi-surface control artifacts for authority surfaces.

Highest-ROI loop upgrades:

  1. Build the report loop around authority, not model output. The loop should start from authoritative artifacts: customer overlay; accommodation catalog; evidence row schema; limitation/accommodation routing rules; renderer/export contract; fixture set.
  2. Use a three-verifier model: schema verifier (is the data contract valid?); source verifier (does every claim row map to source evidence?); renderer verifier (does the final output faithfully represent upstream data?).
  3. Keep bounded pipelines. Do not add a top-level free-form planner agent to the production report generator. Use planners for plans and implementation control artifacts; use deterministic workflows for production report runs.

  5. Add stale/missing-test checks: does the fixture cover unsupported evaluator recommendations? K-12 history-only accommodations? service/referral rows separately from formal accommodations? weak objective data and student self-report-only cases?
  6. Promote customer-specific learnings carefully. Customer overlays belong in overlay artifacts, not global rules, unless the pattern generalizes across customers.

Step 4: Promote Memory Conservatively

Use a three-tier memory path: (1) attempt log: every attempt; (2) candidate memory: lessons from accepted or instructive rejected attempts; (3) promoted skill/rule: only after recurrence or high-severity review.

Promote if:
- lesson appears in >=2 accepted attempts across distinct fixtures, OR
- lesson prevents a P0/P1 review finding, OR
- Eric explicitly asks to make it a reusable rule.

Do not promote if:
- it only fits one customer overlay;
- it contradicts repo-local AGENTS.md;
- it is a positive style preference with no examples;
- it has no expiry/review condition;
- it touches privacy/security behavior without deterministic enforcement.

Step 6: Add Fresh Review Where It Changes Outcomes

Use fresh review when any of the following are true: the authoring loop touched more than one authority surface; the change affects privacy, PII, secrets, or production data; the metric can be overfit; the change modifies a skill, AGENTS.md, or future-agent instruction; the output is customer-facing; the loop was autonomous for more than one iteration.

Step 7: Make Proactive Loops Interrupt Sparse

Every proactive loop should have a written interrupt policy:

alert_if:
  - severity: P0|P1
  - confidence: high
  - evidence_count: ">=2"
  - user_action_required: true
do_not_alert_if:
  - already logged in active thread
  - no new evidence since last check
  - only cosmetic drift
  - blocked by known missing artifact

For Eric's work, high-value proactive monitors are: BC authority docs stale or conflicting; Redactor benchmark drift after GT changes; Linear drift between commits and issues; report-engine fixture coverage gap; PM-Hub research/source-effectiveness degradation; skill/rule files growing without attribution.

Practical Guidance For Future Prompts

Use this when starting an agentic coding loop:

Use the loop-card pattern. First identify the loop type, objective,
verification surface, allowed surfaces, memory rule, and stop conditions.
If the task touches multiple authority surfaces, use multi-surface-agent-harness
before implementation. Do not implement from distributed docs alone.

Use this when launching Redactor optimization:

Run one bounded Redactor optimization attempt. Use the current benchmark as the
verification surface, preserve recall floors, log the attempt and structural
key, and keep/reject based on measured F2 plus entity-type floor checks. Do not
change prompt/model behavior without holdout and explicit approval.

Use this when launching AE report work:

Use the report-grounding loop. Load the authority docs, map affected surfaces,
make one implementation slice, verify schema/source/renderer separately, and
stop on authority conflict or missing source artifact. Do not patch final
rendered strings to hide upstream data-contract gaps.

Final Recommendations

  1. Standardize loop cards now. You already run loops; the cost is coming from inconsistent state, unclear stop rules, and ad hoc memory promotion.
  2. Treat Redactor as the optimization-loop reference implementation.
  3. Treat AE/Bloom/BC as the regulated-document-loop reference implementation.
  4. Treat PM-Hub deep-research as the evidence-loop reference implementation.
  5. Make every loop answer: what changed, what evidence checked it, what was kept or rejected, what memory was created, and why the loop stopped.
  6. Keep autonomy bounded. The best current evidence supports evidence-driven continuation, not open-ended self-direction.
  7. Shift from positive guidance to negative guardrails plus examples.
