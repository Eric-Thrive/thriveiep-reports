BC V3 Graph Engineering: Review and Required Revisions

A read-only review of the proposed graph-engineering strategy for Boston College V3 reports, verified line by line against the live isolated-V3 branch. The strategy is sound. Two parts of it must change before anyone writes code.

BC Architecture Review July 21, 2026 · verified against AE 69790f83c and successor 94640e55f

The research is right about the big thing: BC V3 is already a graph system, and the correct next move is to formalize what exists rather than adopt a graph database or a workflow framework. Every factual claim in the packet held up under direct verification. The two required revisions

These are the changes that must land before implementation starts. Neither one overturns the research conclusion; both narrow the plan.

P1The bank validation graph would rebuild a checker you already own

The proposal is to project the assessment bank into a build-time graph and validate it for dangling IDs, conflicting aliases, duplicate final policies, and stale version pins. Two files in the bank already do most of that work: bank-audit/section5-policy-decision-units.ts checks id uniqueness, orphan bands, orphan subscales, unknown permissions, and interpretation-to-band mismatches (lines 93 to 188), and bank-audit/section5-policy-coverage-audit.ts computes pair coverage, conflicted decision units, and an authority_drift digest gate (lines 62 to 112).

Why it matters. A parallel validator becomes a second referee over the same authority bundle. The existing one is digest-pinned to that bundle; the new one would not be, and the two would drift. Two of the proposed mutation tests would then exist twice with different semantics.

The fix. Extend the existing bank-audit modules in place rather than building alongside them, and add only checks with demonstrated live yield. There is real payoff waiting: the audit turned up a scoring-threshold row, bc-standard-score-100-10-equivalent-at-or-below-93-33 at score-metric-policy.ts:96, that nothing references and that can never fire. The existing checks catch dangling references but not unreferenced rows, so this one is invisible to them. That is one proposed graph check with non-zero yield against current head.

P1Part of the proposed bank graph has no governed data behind it

The target model draws edges from subscale to construct to evidence rule to report placement. But measuredConstructPrimitiveIds is an empty array for every single subscale in the registry (assessment-subscale-registry.ts:302), and the cognitive reporting-domain grouping is a runtime artifact type, not stored bank rows.

Why it matters. Built as drawn, those edges are either silently empty, meaning a validator that always passes and proves nothing, or they get fabricated. Fabrication is precisely the hazard the research itself names: an inferred edge quietly becoming an unsupported conclusion.

The fix. Mark the construct, evidence-rule, and placement edges as absent pending governed bank data plus review. Restrict version one of the bank graph to node types that have real data today: instrument, alias, subscale, metric policy, band, and authority version.

Supporting findings

P2The workflow topology is pinned, not validated

The manifest validator compares a state document against hard-coded constants by exact string equality. Nothing anywhere computes roots, terminals, cycles, reachability, or review dominance over those constants. They happen to be correct right now, which I verified. But a future control revision could introduce a cycle, a second root, or an unreachable card, and every existing check would still pass. Review dominance is enforced only through hand-maintained equality, not derived from the graph. This is the gap the research correctly targets, and it is the reason the P0 slice below is worth doing first.

P2A malformed receipt crashes instead of reporting a bounded failure

In bc-isolated-v3-orchestration-manifest.ts, the review-receipt validator calls value.checks.some(...) at line 278, before the Array.isArray(value.checks) shape check at line 296. A receipt whose checks field is a string or null throws a raw TypeError inside the gate rather than returning a stable issue code. It still fails closed, since a crashed self-test is red, but an unbounded stack trace is exactly the defect class the proposed mutation suite is supposed to eliminate. The fix belongs in the P0 card.

P2Conservation is structural today, not executably verified

The report presents the rendered / review / excluded partition as an existing property. It is emitted, and it holds by control flow because every score either produces a decision or throws. But there is no executable check: the decision-contract validator and the projection-conservation gate are both unbuilt, and the V3_CONSERVATION_FAILED error code is reserved and unused. Separately, the intentionally_excluded branch is defined in the types but no code path ever emits it, because exclusion authority is deliberately refused at runtime.

This raises the value of the proposed evidence-graph receipt, since it would be the first executable cross-artifact conservation check. It also means the receipt must be sequenced with or after the decision-contract validator, so the system ends up with one conservation checker rather than two.

P2"Printed classifications" conflates two different things

Governed printed publisher labels genuinely exist now, for exactly nine adverse bands across eight instruments: "Very Elevated", "Clinically Elevated", "Very High", "Atypical", "Clinically Significant", "Markedly atypical". That is the recent classification-band work. But "Low Average" and "Below Average" are not governed printed labels anywhere. They exist only as an internal band id, a descriptor regex, and a descriptor union type.

That distinction is by design: roughly a hundred cognitive and achievement instruments use anchor-derived numeric thresholds instead of publisher labels. A band node with only one type would generate false "missing instrument-to-band path" findings across all of them, and would misstate the intended precedence rule, which is that exact governed source labels win and absent labels fall through to the numeric evaluator. The graph needs two band subtypes: printed classification band, and numeric anchor band.

P3Three redundant edges, and a lifecycle state that must not be misread

Three orchestration edges into the promotion card are transitively implied by other paths. Harmless today, but the graph compiler has to declare a policy (flag them as intentional, or reduce them), or the "missing dependency edge" mutation test is ambiguous, since removing a redundant edge changes nothing observable.

Separately, on lifecycle state: the branch advanced twice during this review with in-scope remediation commits. The classification-band gate files have an empty diff across the entire range since freeze, with zero touching commits, and the frozen gate now passes six of six. The control document still records the card as GATE_RED with no receipt and no independent review. That is a card legitimately mid-lifecycle, not a regression and not an acceptance. It should not be described as accepted, and any bank-graph work that consumes the new band subtypes should wait for its receipt.

Framework and storage: the research calls are correct

  • Native TypeScript now. Correct. Zero new dependencies, exact fit to existing gate semantics, mutation-testable, and generating the diagram and control tables from one model closes a real doc-drift surface. The renderer function already exists; nothing currently pins the document to it.
  • LangGraph: correctly excluded from report-content decisions, and premature even for the proposed non-authoritative uses. The active control states that orchestration is native delegation with no API scheduler, second state database, or model-authored readiness service for this milestone. Any experiment belongs outside the controlled lane.
  • Temporal: correctly deferred. The trigger conditions the report names, meaning distributed runs, surviving process death, and day-scale human pauses, are genuinely not present yet. Freezing node and edge contracts in TypeScript first is the reusable step either way.
  • Neo4j: correct as disposable, sanitized, analytics-only, never in the generation path. One added constraint: the forbidden-field list applies to any graph store, and no student evidence enters PM-Hub, the research knowledge graph, or any shared store even when described as sanitized. Run-scoped opaque-ID projections only, deleted with the run directory.

No new dependency is approved by this review.

The smallest first step

One card, AE only, roughly one new self-test file plus a bounded edit to the manifest module. It fits inside the repository's three-file and two-hundred-line card rule, and it uses the full receipt tier because it gates a control artifact.

Card GR-1: structural validation of the orchestration graph constants

Gate, written and frozen before any implementation code: a new self-test asserting, over the card, edge, and review constants and over any parsed manifest: exactly one root; exactly one terminal; acyclic; every card reachable from the root; no dangling card or review IDs; every review dependency dominated by its review's predecessor closure; an explicit transitive-redundancy policy; and bounded-failure behavior on malformed receipts, so a non-array checks field yields a stable issue code rather than a TypeError.

Expected red: the structural assertions will pass, because the constants are currently correct. The red must therefore come from the bounded-failure and derived-view assertions. If everything passes before implementation, the gate is invalid under the card state machine and must be revised before freeze.

Mutation fixtures in the same gate: removed edge, added cycle, unreachable card, review-barrier bypass, duplicate terminal, malformed receipt.

Deliberately not in this slice: the evidence-graph receipt, which sequences behind the decision-contract validator and needs an artifact-class decision, and the bank graph, which sequences behind the classification-band receipt and extends the existing audit modules.

Do-not rules

  1. Do not create a second assessment dictionary, alias table, registry, canonicalizer, or a second validator over the Section 5 authority bundle.
  2. Do not emit construct, evidence-rule, or placement edges until governed bank data exists and has been reviewed.
  3. Do not let any graph projection into the report-generation path. Graphs are derived, versioned, digest-bound to their sources, and disposable.
  4. Do not use language models or embeddings to establish instrument identity, metric ownership, thresholds, or policy edges. Candidate links route to review.
  5. Do not put student evidence, including sanitized per-report graphs, into PM-Hub, the research knowledge graph, Neo4j, or any shared store.
  6. Do not reclassify a green-but-unreceipted card as accepted, and do not read a deliberately frozen red gate as a regression.
  7. Do not parallelize metric interpretation through shared mutable state. Metric nodes stay independent and pure.
  8. Do not adopt LangGraph, Temporal, Neo4j, RDF, or SHACL for this milestone.
  9. Do not let an evidence-graph receipt become a new uncontrolled artifact class.
  10. Do not merge donor branches wholesale. Donor facts remain evidence, reconstructed under target-local gates.

Open questions and stop conditions

  1. Source-anchor availability. Does the producer bundle expose opaque region identifiers sufficient for document-to-region-to-occurrence edges, without adding source-sensitive locators to a consumer-side receipt? Stop if the edge would require new source-sensitive fields.
  2. Artifact class. Which existing artifact family owns the evidence-graph receipt: the decision artifact, or the assurance artifact? This needs a root control-card decision. Stop if it would create a new uncontrolled artifact class.
  3. Classification-band closure. The receipt and independent review are in flight in a separate task. Bank-graph work consuming band subtypes should wait.
  4. Control authority. The proposed card touches the control-surface manifest module, and only the root orchestrator may write the control ledger. The card must be claimed there before implementation.

Residual risks

Even with every revision applied, four things stay open. The conservation invariant remains unverified by any executable check until the decision-contract validator lands. Section 5 to Section 6/8 proof promotion has no positive runtime guard until the graph check exists. The intentionally_excluded lifecycle branch is currently unpopulated, so its first real use will need its own gate. And the bank's construct layer is entirely empty, which bounds what any evidence-to-limitation reasoning can honestly claim until governed data is added.
