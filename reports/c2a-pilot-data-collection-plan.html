C2A Pilot Data Collection Plan

PRD v2 · March 31, 2026 · THR-252 · Owner: Eric Falke

Key Messages

  • This is THE data collection plan for the NLU pilot. Every evidence stream -- automated telemetry, student input, advisor feedback, admin rollups -- lives here. No separate "qualitative" vs "quantitative" systems.
  • The SOW requires feedback from both students AND advisors. Version 1 had zero advisor instrumentation. Version 2 adds advisor report tracking, coaching session logs, and structured advisor feedback as P0.
  • Eight workstreams, four build phases, one rollout decision framework. The findings memo doesn't just need data -- it needs a recommendation. Predefined thresholds (expand / modify / hold) make that recommendation evidence-driven.

What are we building?

A measurement layer for C2A that captures how students experience the assessment, how they engage with their Start Here Report, whether advisors access and use the results, and how both groups feel about the process. Elizabeth's evidence protocol defines what the case study needs. This PRD defines what the software captures to produce that evidence.

What changed in v2?

  • New WS7: Advisor Tracking -- report access, coaching session log, time-to-first-view
  • New WS8: Structured Feedback -- student effort score (CES), advisor post-session check-in
  • New Rollout Decision Framework -- three tiers with specific thresholds
  • New Evidence Coverage Audit -- every SOW evidence need mapped to a workstream
Section 2 of 14

The Problem

The SOW requires a findings memo with usage data, insights, and a rollout recommendation. C2A currently captures completion and scores. It does not capture how students experience the assessment, whether they read their reports, or whether advisors use the results. That data cannot be reconstructed after the fact.

ThriveIEP is running a 15-student design partner pilot with NLU. C2A is a strengths-based assessment battery: 17 instruments, 123 items across 3 sprints, generating a personalized "Start Here Report" for each student and their advisor.

Elizabeth's evidence protocol (Notion doc, March 18) defines what the case study needs. Manual items like advisor observations and student interviews are her responsibility. Everything the software can capture, it must. This plan covers:

  • Assessment behavior -- where students linger, quit, or rush
  • Report engagement -- which pages students and advisors view, and for how long
  • Coaching pipeline -- did the advisor see the report? Did coaching happen? Was C2A referenced?
  • Structured feedback -- student effort scores and advisor utility ratings
  • Qualitative touchpoints -- pre/post articulation prompts embedded at natural seams

Section 3 of 14

Goals

G1: Prove C2A changes student self-understanding and advisor behavior

Generate credible evidence that students engage with their assessment results and that advisors incorporate C2A findings into coaching. This is the core claim the NLU case study must support.

G2: Understand how students actually experience C2A

Capture behavioral signals (linger time, save-exit patterns, report section engagement) that reveal where students struggle, disengage, or find value. Not just whether they finished, but how.

G3: Close the advisor feedback loop

Instrument whether advisors access reports, which sections they review, whether coaching sessions occur, and whether C2A results are referenced. Without this, the assessment-to-coaching pipeline is a black box.

G4: Collect structured feedback from both students and advisors

The SOW explicitly requires feedback from both groups. Effort scores, structured check-ins, and open-text prompts must all be captured in the software -- not left to manual collection.

G5: Build reusable analytics infrastructure

The telemetry layer should outlive the NLU pilot. Every future deployment (Tufts, etc.) needs the same usage data. Build once, configure per-deployment via deploymentId.

Section 4 of 14

Non-Goals

Not building Mixpanel

The event schema serves C2A's assessment-to-coaching pipeline. No session replay, heatmaps, or funnel builder.

Not real-time monitoring

Admin views are for periodic review (weekly, per-session), not live ops dashboards.

Not xAPI/Caliper compliant

We adopt the event model (actor/verb/object/timestamp) without the spec overhead. Formal compliance is overkill for 15 students.

Not student-facing analytics

Students see their Start Here Report. They do not see their own usage analytics.

Not automated coaching quality scoring

We instrument whether coaching happened and whether reports were accessed, not whether coaching was good.

Not IRB-grade research infrastructure

This is a design partner pilot, not a clinical trial. Data collection serves product evaluation and the SOW findings memo.

P0 Requirements: Telemetry Must-Have

WS1: Analytics Events Layer

  • c2aAnalyticsEvents table with deploymentId for multi-deployment reuse
  • Client-side useAnalytics() hook: batches events, fires on page unload
  • POST /api/c2a/analytics/events -- batch insert, fire-and-forget
  • 10 event types covering assessment, report, advisor, and feedback actions

Acceptance: Events in DB within 5s. No UX impact (async, non-blocking).

WS2: Report Engagement Tracking

  • usePageTracker on all 5 report pages -- works for both student and advisor views
  • Tracks: entry/exit timestamps, dwell time, scroll depth (25/50/75/100%), accordion opens per skill

Acceptance: Per-student AND per-advisor report page view matrix in admin.

WS3: Assessment Behavior Tracking

  • itemDisplayedAt timestamp + dwellTimeMs per item response
  • Save-exit events with item index, sprint, elapsed time
  • Sprint-level: duration, pause count, items per minute

Acceptance: "Completion topology" showing exact item where each student saved/exited or lingered.

Rollout Decision Framework

The findings memo must end with a recommendation. These criteria make it evidence-driven.

Recommend Expansion

ALL of the following:

  • Assessment completion rate ≥ 80% (12/15 students)
  • Report view rate ≥ 70% of completers (3+ of 5 pages)
  • Advisor report access rate ≥ 60%
  • At least 1 coaching session logged per student
  • Median student effort score ≤ 4 (moderate or below on 1-7 CES)
  • Majority of advisors report "yes, the report helped me prepare"

Recommend with Modifications

ANY of the following:

  • Completion rate 60-79% (identify and fix friction points)
  • Report view rate 50-69% (report UX or notification timing needs work)
  • High effort scores (median > 4) -- assessment too burdensome
  • Advisors not accessing reports -- notification or access flow issue

Do Not Recommend (Yet)

ANY of the following:

  • Completion rate < 60% -- fundamental UX or engagement problem
  • Advisors not logging coaching sessions -- intervention pipeline is broken
  • Student effort scores consistently 6-7 -- assessment too burdensome to scale

Build Sequence

Phase A: Telemetry Foundation

WS1 (events table + hook) → WS3 (assessment behavior) → WS2 (report engagement)

Events layer is the foundation. Assessment behavior and report engagement are the core evidence.

Phase B: Advisor Feedback Loop

WS7 (advisor tracking) → WS8 (structured feedback)

Closes the advisor gap. SOW requires feedback from both students and advisors. This is new in v2.

Phase C: Admin Visibility

WS4 (admin dashboard MVP with advisor columns)

Data without visibility is useless. Dashboard makes all evidence streams queryable and exportable.

Phase D: Engagement Layer

WS5 (notifications) → WS6 (qualitative touchpoints)

Improves the pilot experience. Core evidence collection works without these.

What This Means for Us

Critical Build the events layer before the pilot generates data

Every day of pilot usage without instrumentation is data we can never recover. WS1 must deploy before or immediately after students start Sprint 1.

Critical Resolve blocking questions with Elizabeth this week

The pilot timeline (#1) determines build urgency. Whether NLU advisors have accounts (#2) determines whether Phase B can start. Both need answers before coding begins.

High Advisor instrumentation is not optional

The SOW says "feedback from students AND advisors/coaches." Version 1 had zero advisor data. If the findings memo can't answer "Did advisors use C2A results?", the case study has a hole. WS7 + WS8 are now P0.

High Design the schema for multi-deployment from day one

Adding deploymentId to the events table costs nothing now and prevents a painful migration when Tufts comes online.

Moderate Coordinate WS6 touchpoints with Elizabeth's protocol

The pre/post articulation prompts overlap with her evidence protocol. If she's collecting this via interviews, embedding it in software is redundant. If not, embedding captures responses at maximum salience. Decide before Phase D.

Open Research Questions

  • How should we define "linger time" thresholds? No consensus on meaningful hesitation vs. normal reading for Likert items.
  • What's the right unit of "coaching fidelity" for a 15-student postsecondary pilot? (Meadan et al., 2023 was K-12.)
  • Should pre/post articulation change be measured by text similarity algorithms, human coding, or both?
