Correction Feedback System Guide

How to access and use FP/FN correction signals from the redaction review UI

Redactor Reference v1 (Steps 1-4)

Overview

The redaction pipeline captures user review signals — false positives removed and false negatives added — as structured JSONL for future pipeline improvement. This is a passive accumulation system: data collects automatically, but nothing reads or acts on it without explicit human action.

User Review (text-review-mode.tsx) | | on finalize: collect correction events | v POST /api/feedback ------> corrections.jsonl (append-only, gitignored) | +---------+---------+ | | | v v v jq queries Karpathy learn.md (manual) (future) (future)

Signal Types

SignalMeaningTrigger
fp_removed User removed an auto-detected entity Clicked “remove” on a detection during review
fn_added User manually added a redaction the pipeline missed Added a manual redaction not in the original detections
kept Auto-detected entity survived review unchanged Entity still approved at finalize time (10% sampled)

Accessing the Data

The JSONL file lives at:

server/data/corrections.jsonl

Each line is a standalone JSON object:

{"timestamp":"2026-04-11T14:30:00.000Z","session_id":"abc-123","action":"fp_removed","entity_type":"PERSON","entity_text":"Dr. Smith","confidence":0.45}
{"timestamp":"2026-04-11T14:30:00.000Z","session_id":"abc-123","action":"fn_added","entity_type":"UNKNOWN","entity_text":"Jane Doe","confidence":1.0}
{"timestamp":"2026-04-11T14:30:00.000Z","session_id":"abc-123","action":"kept","entity_type":"DATE_TIME","confidence":0.92}

Quick Inspection Recipes

# Count events by action type
cat server/data/corrections.jsonl | jq -r '.action' | sort | uniq -c | sort -rn

# List all FP removals with entity type and text
cat server/data/corrections.jsonl | jq -r 'select(.action=="fp_removed") | [.entity_type, .entity_text, .confidence] | @tsv'

# List all FN additions (things the pipeline missed)
cat server/data/corrections.jsonl | jq -r 'select(.action=="fn_added") | [.entity_text] | @tsv'

# FP counts by entity type
cat server/data/corrections.jsonl | jq -r 'select(.action=="fp_removed") | .entity_type' | sort | uniq -c | sort -rn

# Top 10 repeated FP texts (systematic pipeline errors)
cat server/data/corrections.jsonl | jq -r 'select(.action=="fp_removed") | .entity_text' | sort | uniq -c | sort -rn | head -10

# Top 10 FN texts (what the pipeline is missing)
cat server/data/corrections.jsonl | jq -r 'select(.action=="fn_added") | .entity_text' | sort | uniq -c | sort -rn | head -10

# Events from a specific session
cat server/data/corrections.jsonl | jq 'select(.session_id=="<id>")'

PII Warning: This file contains real entity text (student names, addresses, SSNs, etc.) from reviewed documents. It is gitignored and must never be committed, copied to shared storage, or included in logs without encryption. The kept events intentionally omit entity_text to minimize PII surface.

Recommendations for Use

Pattern Discovery (Manual)

Start by looking for recurring FP entity types. If PERSON FPs dominate, the pipeline's PERSON detection is too aggressive — likely a post-filter rule or confidence threshold issue. If DATE_TIME FPs dominate, date pattern recognizers may need tightening.

If the same text appears as an FP across multiple sessions, it is a systematic pipeline error worth fixing. One-off FPs are noise.

Feeding into the Karpathy Loop

The /redaction-optimize skill can read this JSONL to prioritize which pipeline axes to tune. FP-heavy entity types suggest precision problems; FN-heavy types suggest recall gaps. The correction data provides real-world signal that the synthetic benchmark cannot.

Future: learn.md Aggregation

Steps 5–7 of the implementation plan (deferred) will aggregate JSONL patterns into docs/redaction-learn.md — a human-readable summary of domain rules and few-shot examples for the LLM verifier. The JSONL file is the raw data asset; learn.md will be the curated view.

Resetting the Data

# Truncate to start fresh
> server/data/corrections.jsonl

There is no rotation or size cap in v1. Monitor file size if running in production for extended periods.
