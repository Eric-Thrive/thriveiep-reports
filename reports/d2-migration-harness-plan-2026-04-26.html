Section 01 of 10

Build a safety harness for moving database tables

Key Messages

  • D2 builds the tooling that makes the upcoming c2a_* table move (D3) safe to execute. Nothing moves yet; this is the rehearsal infrastructure.
  • Generalized, not throwaway. When bloom_*, users, or organizations get split next year, D4 and D5 import this same harness instead of writing one-off scripts.
  • Catches a known Drizzle bug class automatically. Cross-schema foreign keys silently ship as malformed SQL with current Drizzle versions; D2's two-layer linter blocks them in CI.

What did we decide to build?

A new workspace package called @accommodation-engine/migrate with a typed API, a CLI, and CI integration. Six public functions. Three flows (CI gate, integration test, operator dry-run). All dev-tool only — never imported into the running server.

Why is it worth its own plan?

D3 is the risky part: moving 24 production tables out of public into a dedicated c2a schema. If we bundle the harness into D3, every harness question gets re-litigated under deadline pressure. Splitting the work means D3 becomes a small concrete task ("here's the table list, here's the harness, here's the runbook") instead of "design the tools and use them in the same PR."

What does it cost?

One new workspace, four dev-only dependencies (testcontainers, pg-query parser, commander, pg). Two new CI jobs. The plan is 13 new files + 4 modified files. No production code changes. No schema changes. No data changes.

Section 02 of 10

Why we need this — three real problems

The master roadmap (Plan D = Schema Split & CI/CD) defines four specific problems that D2 must solve before D3 can safely run.

Problem 1: Drizzle would destroy data

If you ask Drizzle to move a table from public to a new schema, it generates DROP TABLE + CREATE TABLE — wiping every row. The correct approach is hand-written ALTER TABLE ... SET SCHEMA, which moves the table in place. D2 ships a generator that produces this SQL correctly, plus a companion ALTER FUNCTION ... SET SCHEMA for trigger functions (which Postgres does NOT move automatically with the table).

Problem 2: Drizzle has documented bugs in cross-schema foreign keys

Drizzle issues #636, #440, #1568, #890 all describe the same failure mode: the SQL Drizzle generates for a foreign key that crosses schema boundaries looks fine but the constraint that actually applies in Postgres is broken or missing. CI never sees it because the file scan looks at the source SQL, not the applied state. D2 ships a two-layer linter that catches both layers.

Problem 3: No zero-downtime strategy exists

The moment ALTER TABLE ... SET SCHEMA runs, the old table location stops working. Any container still running the old code crashes. The expand/contract pattern with views as a backward-compatibility bridge solves this — but only if we generate the view SQL correctly. D2 ships the generator. (D3 owns the runtime cutover sequencing — when readers swap, when views drop.)

Problem 4: We've never moved a table at this scale

The c2a_* surface is 24 tables, all referenced by application code in dozens of places. We need to rehearse the entire move — forward and rollback — in an environment where mistakes cost nothing. D2 ships the rehearsal mechanism.

Section 05 of 10

Two-layer FK linter — the most novel piece

Cross-schema foreign keys are where Drizzle bugs live. A single-layer check isn't enough because the failure mode is "SQL file looks correct, applied constraint is broken." We need both eyes.

Layer 1: Static AST scan (every PR, <30s, no Docker)

What it does

Parses every .sql file in packages/db/migrations/ with pg-query-emscripten. Walks the AST for FOREIGN KEY ... REFERENCES <schema>.<table> nodes where the schema differs from the owning table's schema. Allowlist mechanism: a comment -- fk-linter-exempt: THR-NNN <reason> on the line before the constraint suppresses the warning.

What it catches

Plans that introduce cross-schema FKs without explicit hand-written approval.

What it misses

Drizzle bugs where the file looks fine but the applied constraint is broken. That's why we need Layer 2.

Layer 2: Runtime introspection (during shadow split)

What it does

After migrations apply to ephemeral postgres, queries information_schema.table_constraints joined with information_schema.referential_constraints joined with pg_namespace. Extracts every actually-applied FK boundary. Compares against a typed allowlist at packages/migrate/src/cross-schema-fk-allowlist.ts.

The allowlist itself

Each entry requires thrTicket, addedDate, justification, and reviewDate. CI fails if any entry's review date is more than 180 days old (mirrors D1's allowlist sunset pattern). D2 ships an empty array — no cross-schema FKs exist yet.

Failure ergonomics matter

When the linter finds an unexpected FK, the error message prints the exact JSON snippet to add to the allowlist file (or the SQL to remove the FK). Engineers don't have to guess what to do.

Why this isn't overkill: Drizzle issues #636, #440, #1568, #890 all describe the same class of bug. The harness's whole point is to catch these before D3 hits them in production.
Section 06 of 10

Shadow split — the rehearsal that makes D3 safe

Before D3 moves a single real table, the harness rehearses the entire move + rollback against an ephemeral postgres in CI. If the move doesn't round-trip cleanly, the PR fails.

The rehearsal sequence

  1. Spin up postgres:16-alpine via testcontainers
  2. Apply 0000_canonical_baseline.sql + 0001_c2a_session_uniqueness.sql
  3. Snapshot information_schema as baseline
  4. Generate the forward SQL via generateSetSchemaMigration and apply it
  5. Run runtime FK lint
  6. Snapshot as afterForward
  7. Verify SELECT works through the public-schema view
  8. Apply the reverse SQL
  9. Snapshot as afterRollback
  10. Assert equivalentSnapshots(baseline, afterRollback) === true
  11. Tear down the container

Two modes — both first-class

D2 default create-synthetics — the harness creates 4 synthetic tables before the move, exercises the pattern, asserts rollback parity. Self-contained. CI never touches real table names.

D3 entry point use-existing — the harness operates on tables already present from canonical baseline + 0001 (D3's real c2a_* table list). Skips synthetic creation.

Why both modes matter: Plan-harden surfaced that the original design only had create-synthetics mode — meaning the harness as designed could not do D3's actual job. The adversarial reviewer caught this; without the fix, D3 would have hit the wall mid-execution.

What "equivalent" means precisely

A custom comparator (equivalentSnapshots) compares only the structural fields that matter: tables, columns, constraints, indexes, views, triggers, sequences (excluding lastValue, which advances during synthetic inserts; excluding oid, which is non-deterministic). Definitions are passed through canonicalizePgDef first to absorb cross-version pretty-printing differences from pg_get_constraintdef.

Section 07 of 10

Three ways the harness gets used

The same code path serves three different callers — CI, dev, and D3's production runbook — by varying inputs.

Flow A — CI gate (every PR, <30s, no Docker)

The static FK linter runs on every PR as a fast gate. If anyone introduces a cross-schema FK without an allowlist comment, the PR turns red. No DB, no Docker, just file scan + AST walk.

migrate-harness-static
  └─ npm run migrate:lint-fks --json
       └─ lintCrossSchemaFksStatic("packages/db/migrations/")
       └─ exit 0 or exit 1 with violations JSON

Flow B — Shadow-split integration (paths-filter PR + nightly, ~120s)

The full shadow split runs only when migration code changes (paths-filter on packages/migrate/** or packages/db/migrations/**) plus once nightly as a regression gate. Docker required; testcontainers handles port allocation and cleanup.

migrate-harness-shadow
  └─ npm run migrate:shadow-split -- --mode create-synthetics --json
       └─ runShadowSplit({...defaults, teardown: true})
       └─ emit ShadowSplitResult JSON; non-zero exit on phase fail

Flow C — Operator dry-run (local or D3 runbook, unbounded)

Engineers (and D3's runbook) call the CLI directly to dry-run a specific migration before touching production. Same code path as Flow B; different entry.

$ npx @accommodation-engine/migrate dry-run \
    packages/db/migrations/0002_c2a_schema_split.sql \
    --baseline packages/db/migrations/meta/0002_snapshot.json
  └─ runDryRun({migrationPath, expectedBaseline, prerequisites})
  └─ print human-readable diff + lint results
  └─ exit 0 / 1

Why structured results matter

Every flow returns the same shape: SchemaSnapshot, StaticLintResult, RuntimeLintResult. D3's runbook calls runShadowSplit against the real c2a_* table list, then runDryRun against the actual schema-split migration. The structured results go straight into D3's PR description as evidence — no manual log parsing.

Section 09 of 10

What's intentionally NOT in D2

Discipline about scope is what makes D2 shippable. Every item below was considered and explicitly deferred — not forgotten.

Belongs to D3

  • The actual c2a_* schema split. D2 builds the harness; D3 uses it.
  • drizzle.config.ts per-module split (FIX-4). Only matters once real pgSchemas exist in schema.ts.
  • Views cleanup job. D3 removes the public-schema views after all readers and writers cut over to the new schema.
  • pgSchema('c2a') declarations in schema.ts. Drizzle-side change that goes with D3.
  • Real Aptible execution of any migration. D3's runbook handles production apply; D2's harness only ever touches ephemeral postgres.
  • Runtime cutover sequencing for the views bridge. When readers swap, when views drop, the verify-no-public-readers scan — all D3 territory. D2 only ships the SQL primitive.

Belongs to D4 / D5 (future splits)

  • bloom_* table split, users split, organizations split. These will reuse this harness rather than re-invent.

Orthogonal to Plan D entirely

  • pnpm migration. Separate concern; can happen before, during, or after Plan D without coupling.

Inherited from c2a-db-hardening, deferred to D3's prod-execution wrappers

  • Time-window apply guard (C2A_REMEDIATE_NOT_BEFORE ISO-8601 cutoff). D2 has no production side. D3 must re-add at the invocation layer.
  • SSL sslmode rewrite. D2 uses testcontainers (no SSL); operators rely on node-postgres native handling.
Why these are explicit non-goals, not silent omissions: Plan-harden caught both as "replacement-utility silent-drop" findings — when generalizing scripts into a harness API, edge-case guards from the original scripts disappear unless explicitly named. Documenting them as non-goals makes them carry-forward checklist items for D3.
