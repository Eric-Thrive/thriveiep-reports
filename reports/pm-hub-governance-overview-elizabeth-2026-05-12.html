Section 01 of 08
PM-Hub overview

PM-Hub becomes the operating system for ThriveIEP product work.

The simple idea: PM-Hub is where plans, decisions, research, implementation notes, and shareable artifacts live first. Google Docs is the polished sharing format when a stakeholder needs to review or comment outside the repo.

For Elizabeth: this means less scavenger hunting, clearer handoffs, and a consistent path from internal thinking to stakeholder-ready docs.

Section 02 of 08

What changes in practice?

Source of truth

Work starts in PM-Hub

Product specs, decision records, plans, and research summaries live as structured files with clear owners, dates, tags, and status.

Governance

The repo warns early

New guardrails catch missing metadata, messy filenames, stale exports, and orphaned deliverables before the system drifts.

Sharing

Docs leave intentionally

Google Docs exports are on demand, one-way, and tracked in frontmatter so everyone can see what has been shared.

Section 03 of 08

The workflow is designed around PM decisions, not file maintenance.

1

Create

A plan, spec, decision record, research brief, or meeting artifact is written in PM-Hub.

2

Tag

Metadata says what it is, what domain it belongs to, and whether it is draft, active, accepted, or complete.

3

Check

Soft lint warnings catch missing fields, invalid tags, naming issues, and version-history problems.

4

Discover

The artifact page and knowledge graph make the work searchable by domain, status, tag, and relationship.

5

Share

When needed, selected PM-Hub files export to Google Docs or publish as HTML reports.

Section 05 of 08

Google Docs integration is intentionally one-way.

PM-Hub remains the source of truth. Google Docs is a stakeholder-friendly copy when people need comments, review, or a familiar document link.

StepWhat happensWhy it matters
Choose one file Export starts from an explicit PM-Hub markdown path, not a bulk sync. Prevents accidental sharing and keeps sensitive content deliberate.
Pick a Drive folder The export skill browses Google Drive and stores the chosen folder label and ID. Future re-exports know where the stakeholder copy lives.
Create or update the Doc Markdown becomes Google Docs content; frontmatter stays in PM-Hub as metadata. Elizabeth gets a readable doc without losing PM-Hub structure.
Track freshness PM-Hub records the Docs URL and last exported date. The weekly digest can flag stale exports when the source changed later.
Section 06 of 08

The governance layer is a soft guardrail, not a blocker.

What gets checked

  • Missing or invalid frontmatter
  • Required fields such as title, tags, status, created, and updated
  • Tags that are outside the PM-Hub taxonomy
  • Filenames with spaces, backups, or duplicate-style suffixes
  • Version history entries that are missing dates or notes

What it does not do

  • It does not reject commits
  • It does not rewrite old files in bulk
  • It does not auto-export anything to Google Docs
  • It does not sync Google Docs content back into PM-Hub
  • It does not expose raw private data in reports

Section 08 of 08

The rollout is deliberately incremental.

First

Measure the baseline

Run the linter across PM-Hub and report the current state without forcing a cleanup sprint.

Next

Turn on routine visibility

Add changed-file warnings after commits, weekly health reporting, and the artifact discovery page.

Then

Use exports selectively

Export high-value docs to Google Docs when stakeholder review is needed, with freshness tracking.

Bottom line: PM-Hub keeps the product memory durable. Google Docs makes the right pieces easy to review with the right people.
