Session handoffs — how memory persists

The problem

Claude sessions have finite context. Without structured memory, every conversation re-explains the same facts — expensive, slow, error-prone.

The pattern (DR-004)

  • End of session: the session-handoff skill writes a structured entry — focus, decisions, artifacts, gotchas, pickup items
  • Rolling buffer: last 5 sessions live in handoff.md; older ones auto-archive
  • Start of session: Claude reads handoff.md + priorities.md + queries the KG before responding

Why it matters for us

S365 (today) inherited unresolved threads from S364, S363, S259 automatically. Decisions made 2 months ago stay in scope. Continuity is engineered, not remembered.

Slide 2 of 2 — The intelligence layer

Knowledge graphs + skills make PM-Hub queryable, not just searchable

Three KG layers give Claude structural context without reading every file. A library of skills automates the recurring workflows we’ve refined over 365 sessions.

Knowledge graphs (3 layers)

Product KG — pm-hub-kg

What: Decision records, docs, sessions, Linear issues with explicit + semantic links

How: Structural graph (frontmatter refs) + ChromaDB vector search (sentence-transformers, local)

Tools: query_graph, traverse, find_stale, find_orphans, suggest_gaps

Use: “Which DRs touch FERPA?” · “What docs reference THR-222?”

Code KG — code-graph

What: AST-extracted call graphs for the Accommodation Engine and Redactor codebases (3,372 nodes, 6,606 edges)

How: Auto-rebuilt by post-commit hook; ts-morph + Neo4j

Tools: impact_analysis, find_callers, list_routes, pipeline_overview

Use: Refactor blast radius before touching code

Research KG — neo4j-kg + wiki

What: 489 papers, ~9.2k nodes; compiled into 653 wiki articles (38 topics, 85 entities, 146 concepts)

How: /deep-research ingests papers; auto-compiles into topic syntheses with health/confidence labels

Use: SDT, EF coaching, PII detection, assessment instruments — literature on tap
