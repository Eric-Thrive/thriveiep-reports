Architecture Overview

Post-Secondary Document Review Generator

Updated architecture and build-status overview for the Boston College CFLC pre-interview document review packet. The production Post-Secondary module is now implemented through server-authoritative export preflight.

Updated May 14, 2026 · ThriveIEP Accommodation Engine
Section 1 of 18 — Architecture Principle

Same principle as Bloom, plus one new layer

Predictable work uses fixed rules. Judgment work is supported by a hosted language model. Uncertain results go to a human reviewer. And now: customer overlays control what the rules say.

Fixed rules

Same input, same result

Evidence compilation, date math, schema checks, export gates, accommodation-to-evidence linking, active-version resolution, and Drive destination policy validation.

Used for everything that can be tested and trusted without judgment.
Hosted model

Language requires judgment

Interpreting unfamiliar document language, proposing candidate mappings, and drafting review text only where deterministic rules are not enough.

The model proposes. Rules, overlays, and humans confirm.
Human review

Routing for uncertainty

Low confidence, conflicting evidence, out-of-catalog accommodations, missing documentation, and anything the rules can't resolve.

CFLC staff see flags, not final decisions.
Current status: the Post-Secondary package, BC overlay path, Google Doc projection, quality gates, durable export intents, reviewer acknowledgements, trusted run-input loading, and Drive policy preflight are implemented behind server gates.

Section 3 of 18 — Relationship to Bloom

A sibling, not a fork

Post-Secondary shares the same infrastructure layer as Bloom but owns its own rules. They never import each other's code.

Shared with Bloom

Report engine contracts: evidence packets, determinations, render plans, quality reviews
Staged pipeline: fixed rules first, model only where needed, human review for uncertainty
Google Docs export infrastructure: Drive policy validation, durable intents, acknowledgements, audit logging, idempotent retry
Redactor for privacy and text preparation
Audit bundle pattern with full provenance

New for Post-Secondary

Customer overlays: BC controls its own catalog, labels, scope filters, and wording rules
Functional limitation index with stable IDs linking limitations to accommodation evidence
Four required accommodation section slots that every customer must fill
Run modes: evidence-only, pre-interview, complete, full report
Staff-facing Google Doc packet first. The legacy in-app viewer is not on the production path
Boundary rule: Post-Secondary imports from the shared report engine. It never imports from Bloom, and Bloom never imports from it.

Section 7 of 18 — Staged Pipeline

Accommodations and export

Evidence-linked candidates, quality gates, then a Google Doc.

1
Parse documentsFixed
2
Extract evidence claimsModel
3
Build functional limitationsFixedModel
4
Apply customer scopeOverlay
5
Map accommodation candidates Fixed Model Each candidate gets a stable ID (like AC-001), links back to limitation IDs, and a support status. A recommendation alone can't create a supported status. It needs current evidence and customer-scope approval.
6
Quality gates + export Fixed Plain-language checks, missing-source warnings, privacy scan, citation verification, Google Doc projection, durable export intent, reviewer acknowledgements, and Drive destination policy preflight.
Critical guardrail: A clinician recommendation, a student request, or a historical IEP accommodation can create a candidate row. But none of them alone can make the system say "this is supported." Current evidence linked to a functional limitation is required.
Section 8 of 18 — Report Structure

Four accommodation slots, always present

Every projection keeps the four section slots available. BC controls the labels, allowed entries, implementation-note wording, and resources through the overlay; future customers can rename and filter them without changing the engine.

Academic Accommodations

Testing accommodations, deadline flexibility, note-taking support, attendance policies. The core academic adjustments that address documented functional limitations.

BC labels and allowed entries come from the overlay catalog.

Instructional / Program Accommodations

Course-load adjustments, program-specific modifications, classroom environment changes. Things that affect how the student experiences instruction.

Separate from academic testing accommodations by design.

Auxiliary Aids & Services

Assistive technology, accessible materials, text-to-speech, recording permissions. Tools and resources that support access.

BC's overlay can restrict to their available technology catalog.

Non-Accommodation Supports & Referrals

Tutoring center referrals, study skills workshops, counseling. Helpful resources that aren't formal accommodations and don't require the same evidence standard.

Never rendered as formal accommodations. These are suggestions, not determinations.
Section 9 of 18 — Run Modes

Different outputs from the same pipeline

The same evidence engine can produce different deliverables depending on where BC is in their process. The first two profiles now exist in the package-level projection path.

Implemented in fixture path

Evidence Generator

Organizes documentation into evidence, functional limitations, candidate rows, gaps, and interview prompts. No final recommendations. Useful before BC has a student interview.

Implemented in fixture path

Pre-Interview Packet

The BC staff Google Doc projection for the student's first meeting. Includes evidence-generator content plus preliminary matrix rows. Avoids all final-decision language.

Future

Complete Packet

After the student interview and CFLC staff judgment. Adds interview findings, staff rationale, selected outcomes, and unresolved gaps. The retained record.

Future

Full Student Report

A complete student-facing report with explanatory prose, recommendation language the customer has approved, and full accommodation detail.

Section 10 of 18 — Audit & Trust Layer

What the backend produces besides the Google Doc

The staff packet is one output. The audit artifacts are the trust layer. They show what the system concluded, why, where it was uncertain, and which gated version produced the output.

Evidence Packet

Every claim extracted from the uploaded documents, with dates, source locations, and confidence levels.

Functional Limitation Index

Stable IDs (FI-001, FI-002...), plain-language labels, evidence links, mapping method, customer-scope status, and QC status for each limitation.

Accommodation Evidence Matrix

Every candidate row linked to limitation IDs, with support status, evidence strength, and gap reasons.

Quality Review Summary

Flags for missing sources, unsupported statuses, plain-language violations, privacy concerns, and stale evidence.

Render Plan

Section-by-section blueprint for the Google Doc, including which rows, determinations, and flags appear in each section.

Audit Bundle

Run ID, case ID, pack version, overlay version, rules fired, every matrix row with provenance, export intent state, acknowledgement state, and Drive policy validation metadata.

Every status in the staff packet can be traced back to source evidence, mapping rules, and overlay configuration through these artifacts.

1 Does the evidence-first approach match how CFLC staff actually review documentation?
2 Are the four accommodation slots the right structure for BC's pre-interview packet?
3 Does the "evidence generator" mode (documentation prep without recommendations) have standalone value?

Section 13 of 18 — Discussion
1

Evidence-first review workflow

Does separating "what's in the documentation" from "what accommodations to consider" match how CFLC staff actually work?

Listen for

Whether staff currently read all documents before forming accommodation ideas, or whether they jump to accommodations while reading. The implemented path keeps functional limitations separate from accommodation candidates. Does that match reality?

Watch for

Moments where the clean separation feels artificial. If a staff member always thinks "I see X limitation, so I know Y accommodation" simultaneously, the two-step pipeline might feel like extra work rather than clarity.

Decide today

Should the Google Doc show functional limitations in one section and accommodation candidates in another? Or interleave them so each limitation has its candidates right below it?

Section 14 of 18 — Discussion
2

Four accommodation slots for BC

The system keeps four configurable section slots. Do they match how BC thinks about supports?

Listen for

Whether BC uses these same four categories internally, or whether their mental model is different. Maybe three buckets, maybe five. The slot structure should match their workflow, not ours.

Watch for

Items that don't clearly belong in one slot. Is "priority registration" academic or instructional? Is "reduced course load" an accommodation or a program modification? These edge cases reveal whether the categories are clear enough.

Decide today

Should BC's non-accommodation supports section include referral contact info and next steps, or just the name of the resource? How much logistics detail belongs in the pre-interview packet?

Section 15 of 18 — Discussion
3

Evidence generator as standalone tool

Before a student even arrives: could a "documentation prep" output be useful on its own?

Listen for

Whether staff currently spend significant time reading and summarizing uploaded documents before meeting a student. If yes, an evidence-only output that organizes claims, dates, and gaps saves prep time even without accommodation candidates.

Watch for

Whether "evidence without recommendations" feels incomplete or confusing. If staff expect the system to always suggest accommodations, a documentation-only mode might feel like a half-finished product rather than a prep tool.

Decide today

When documents are missing student narrative, should the system generate interview prompts based on evidence gaps? The current projection can carry prompts; BC should decide how assertive they should feel.

Section 16 of 18 — Discussion
4

Quality flags and review triggers

The system flags uncertainty. Which flags should pause the workflow, and which are just notes for staff?

Listen for

Which types of problems staff catch most often in their current process: conflicting dates? Missing evaluations? Stale records? Recommendations without supporting evidence? This tells us which flags matter most.

Watch for

Flag fatigue. If every row has a yellow warning, staff will start ignoring them. The system needs to distinguish between "this needs your attention before the meeting" and "FYI, this is slightly old."

Decide today

Which flags should block export, which should require acknowledgement, and which should remain informational? The server now supports durable acknowledgement before Drive readiness.

Section 17 of 18 — Discussion
5

Customer overlays: "your rules, your labels"

The overlay concept means BC controls their own configuration. Is that clear enough to sell?

Listen for

Whether "your rules control what the system does" resonates with BC, or whether it sounds like we're asking them to do configuration work. The pitch should feel like customization, not setup burden.

Watch for

Assumptions about what BC can customize versus what's fixed. They should understand: they can rename sections, restrict the catalog, change wording, and tune resources. They cannot bypass evidence-to-accommodation linking, privacy gates, or Drive destination governance.

Decide today

What does BC's initial accommodation catalog actually look like, and which entries should be standard versus fallback-to-Thrive-default? This answer controls the first production overlay seed.
