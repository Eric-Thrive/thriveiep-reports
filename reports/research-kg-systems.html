System Overview

Three layers, strictly ordered — each layer reads the one below and never writes to it.

┌─────────────────────────────────────────────────────┐ │ Layer 3 — Knowledge Graphs │ │ _KNOWLEDGE_GRAPH.json | Neo4j (Railway) │ │ ChromaDB semantic index | BM25S keyword index │ │ pm-hub-kg MCP (query_graph, traverse, find_stale) │ └───────────────┬─────────────────────────────────────┘ │ indexes / embeds ┌───────────────▼─────────────────────────────────────┐ │ Layer 2 — Research Wiki │ │ research/wiki/ (topics/ entities/ concepts/ papers/) │ LLM-maintained markdown articles (656 total) │ │ compiled from raw findings via /research-wiki skill │ └───────────────┬─────────────────────────────────────┘ │ compiles from ┌───────────────▼─────────────────────────────────────┐ │ Layer 1 — Raw Findings │ │ research/findings/*.json (118 files, immutable) │ │ research/sessions/*.md (81 session logs) │ │ Source of truth — LLM reads but never modifies │ └─────────────────────────────────────────────────────┘

Why separate layers? Raw findings are immutable so claims can always be traced to their source. The wiki synthesizes across sessions. The KG enables fast retrieval without re-reading every file. The layers can be rebuilt independently when schemas change.

Layer 1 Raw Findings

Location

  • research/findings/ — 118 JSON files, one per research session
  • research/sessions/ — 81 Markdown session logs (human-readable companion)

Findings JSON Schema

Each findings/*.json is a ResearchFindings Pydantic model:

FieldTypeDescription
session_idstringSlug identifying the research session
paperslist[PaperFinding]Papers found, with DOI, year, citation count, abstract
claimslist[ClaimFinding]Extracted claims with paper provenance
conceptslist[ConceptFinding]Concept co-occurrences across papers
benchmark_resultslistModel performance on tasks
appraisalslist[AppraisalResult]Critical appraisal (RoB, GRADE)

Immutability rule: Never modify findings files after creation. If a session's evidence is superseded, the wiki article gets a superseded_by: frontmatter field — the raw finding stays intact.

Aliases Registry

research/aliases.json — 1,653-line canonical name registry. Maps variant names (e.g., "RoB 2", "Cochrane RoB", "bias assessment") to one canonical concept slug. The wiki compile router uses this to avoid duplicate concept pages.

research/pending-aliases.json — aliases awaiting human approval before promotion.

Compile → reindex_with_health.py

The /research-wiki compile skill reads raw findings and writes/updates wiki articles. After any compile, run reindex_with_health.py to regenerate index.md and the BM25S index:

cd PM-Hub && python3 scripts/reindex_with_health.py

This script: (1) parses YAML frontmatter from all articles, (2) computes a health status for each, (3) applies retention decay when decay_class: is set, (4) builds the BM25S keyword index, (5) writes wiki/index.md.

Article Frontmatter (key fields)

title: "Article Title"
type: topic          # topic | entity | concept | paper
provenance: primary  # primary | derived | mixed
confidence: high     # high | medium | low
papers: 12
sources:
  - ../findings/2026-04-06-instrument-audit.json
last_session: 2026-04-06
session_slugs: [instrument-audit]
decay_class: slow    # slow (90d) | medium (45d) | fast (14d) half-life
relationships:
  - type: evidences
    target: ../entities/gliner.md
  - type: operationalizes
    target: ../concepts/pii-detection.md

Health Status Logic

StatusCondition
stubconfidence: low AND papers = 0
agingEvidence beyond domain freshness window (or freshness_score < 0.5 if decay_class set)
primaryHas sources:, no derived_from:
derivedHas derived_from:, derivation_depth ≥ 1
mixedHas both sources: and derived_from:

Typed Relationships

Articles use relationships: instead of the deprecated flat related: list. Once relationships: is populated, related: is frozen — all new cross-references go into relationships:.

TypeDirectionMeaning
contradictsbidirectionalClaims in conflict
evidencespaper/entity → topic/conceptEmpirical support flows to target
operationalizesentity → conceptConcrete implementation of a concept
usestopic/entity → entityDepends on or applies
extendsany → anyBuilds on without contradicting

Supersession Fields

When a claim is superseded by newer evidence:

superseded_by: ../concepts/newer-claim.md
valid_until: 2026-03-15     # date this claim became stale
supersedes: ../concepts/older-claim.md

Run scripts/backfill-valid-until.py to automatically write valid_until: dates to articles that have superseded_by: but are missing the date anchor.

Layer 3 Knowledge Graphs

Two separate KGs at different granularities. They do not overlap — the bridge is the compiled_from: frontmatter field, which links articles → findings JSONs → Neo4j paper DOIs.

Article KG (_KNOWLEDGE_GRAPH.json) Paper/Claim KG (Neo4j, Railway) ───────────────────────────────────── ────────────────────────────────── Wiki article nodes Paper nodes Typed article-article edges Claim nodes DR / C2A doc / decisions nodes Concept co-occurrence edges Structural + semantic search Literature queries, GRADE/RoB Exposed via pm-hub-kg MCP Exposed via neo4j-query.py Always available (local file) Requires Railway connection

Critical constraint: Article-level relationships: frontmatter is not ingested into Neo4j. Those edges belong in _KNOWLEDGE_GRAPH.json only (built by build-knowledge-graph.py). See kg-architecture.md in the deep-research skill references.

KG A — Article KG (_KNOWLEDGE_GRAPH.json)

Built by: scripts/build-knowledge-graph.py
Covers: All .md files across PM-Hub (wiki articles, DRs, C2A docs, decisions, etc.)
Node attributes: title, domain, confidence, last_session, valid_until, stale, superseded_by, session_slugs, compile_date
Edge types: wiki_related, supersedes, compiled_from, plus all typed relationships: types

cd PM-Hub && python3 scripts/build-knowledge-graph.py

Also runs automatically via post-commit hook (incremental).

KG B — Paper/Claim KG (Neo4j)

Built by: scripts/neo4j-ingest.py
Covers: Raw research findings only — papers, claims, concepts, appraisals
Requires: NEO4J_URI, NEO4J_USERNAME, NEO4J_PASSWORD in .env (Railway-hosted, may be suspended)

cd PM-Hub && source venv/bin/activate
python3 scripts/neo4j-ingest.py findings/<session>.json

KG C — Semantic Index (ChromaDB)

Built by: scripts/ingest-chromadb.py
Covers: All PM-Hub markdown + JSON, chunked and embedded with sentence-transformers
Location: _chromadb/ (persistent local)

# Full rebuild (weekly or after major content changes)
python3 scripts/ingest-chromadb.py --full

# Incremental (auto via post-commit hook)
bash scripts/incremental-semantic.sh

KG D — BM25S Keyword Index

Built by: scripts/reindex_with_health.py (as part of wiki reindex)
Covers: Wiki articles only
Location: research/wiki/.bm25-index/ (bm25s native binary format)
Purpose: Exact-match rescue for technical acronyms (WISC-V, GLiNER, Presidio) that semantic search misses. Fused with ChromaDB results via Reciprocal Rank Fusion (RRF).

Staleness check: wiki/.compile-state.json records bm25_build_date. If the BM25 index is older than the last compile, the query path warns and falls back to ChromaDB + graph only.

Search Pipeline

When query_graph(question) is called via the pm-hub-kg MCP:

1
Embed question → ChromaDB vector search (top 10)
2
BM25S keyword search (top 10) — wiki articles only
3
RRF fusion: score = Σ 1 / (60 + rank_i)
4
Graph expansion: for each top result, follow _structural_links() (outbound relationships, compiled_from, superseded_by)
5
Apply source effectiveness boost (from telemetry/search-effectiveness.json)
6
Apply recency bonus (git mtime vs frontmatter updated: date)
7
Return ranked results with trust banner, health status, typed edges

Mode parameter: hybrid (default) — ChromaDB + graph expansion  |  vector — pure ChromaDB similarity

Rebuild Workflows

After a /deep-research session

# 1. Research skill auto-saves findings to research/findings/
# 2. Run wiki compile (incremental — only new sessions)
/research-wiki compile --session <slug>

# 3. Rebuild wiki index + BM25S
python3 scripts/reindex_with_health.py

# 4. Rebuild article KG (also regenerates auto-generated indexes)
python3 scripts/build-knowledge-graph.py

# 5. Ingest papers/claims into Neo4j (if Railway is available)
source venv/bin/activate && python3 scripts/neo4j-ingest.py findings/<slug>.json

# 6. Commit (triggers incremental ChromaDB update via post-commit hook)
git add -A && git commit -m "[Session N] Research: <topic>"

Full semantic rebuild (weekly or after major content change)

cd PM-Hub && source venv/bin/activate
python3 scripts/ingest-chromadb.py --full

After schema migration (new frontmatter fields)

# Backfill provenance fields
python3 scripts/backfill_provenance.py

# Backfill valid_until dates (only if supersession pairs exist)
python3 scripts/backfill-valid-until.py

# Rebuild everything
python3 scripts/reindex_with_health.py
python3 scripts/build-knowledge-graph.py

Key Constraints

1. Layer 1 is immutable. research/findings/ files are never modified after creation. Superseded evidence is annotated in the wiki, not deleted from findings.

2. Article edges stay in the article KG. relationships: frontmatter is parsed by build-knowledge-graph.py into _KNOWLEDGE_GRAPH.json. It is not ingested into Neo4j.

3. yaml.safe_load required for frontmatter. The relationships: block is a YAML list-of-mappings. Hand-rolled line parsers cannot read it.

4. bm25s ≠ rank_bm25. Incompatible APIs and index formats. Always use bm25s — community drop-in-replacement claims are incorrect.

5. _KNOWLEDGE_GRAPH.json nodes are flat. No "attributes" wrapper. Access fields directly from the node dict (e.g., node["last_session"], not node["attributes"]["last_session"]).

6. Concept creation requires multi-session evidence or explicit cross-topic reuse. Local session density alone (many claims in a single session) is not sufficient to create a concept page.
